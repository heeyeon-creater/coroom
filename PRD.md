# PRD: coroom (회의실 예약 시스템)

## 1. 개요

| 항목 | 내용 |
|---|---|
| 프로젝트명 | coroom |
| 한 줄 정의 | 사내 회의실(1~6번)의 예약 현황을 한눈에 보고, 빈 시간대를 클릭해 바로 예약할 수 있는 웹 서비스 |
| 문제 상황 | 기존에는 엑셀 시트(`회의실목록`, `예약현황`)로 예약을 수기 관리 → 실시간 현황 파악 불가, 중복 예약 가능성, 조회/취소 불편 |
| 해결 방향 | 회의실별 시간표를 그리드로 시각화하고, 빈 칸 클릭 → 즉시 예약 폼 입력의 원클릭 예약 플로우 제공 |
| 데이터베이스 | Supabase (PostgreSQL + Realtime + Auth) |

## 2. 목표

- 회의실 예약 현황을 **하나의 화면에서** 실시간으로 확인
- 예약 절차를 **"빈 시간 클릭 → 정보 입력 → 완료"** 3단계로 단순화
- 이중 예약(더블 부킹)을 시스템 차원에서 방지
- 기존 엑셀 관리 대비 조회/취소/변경 속도 개선

## 3. 타겟 사용자

- 사내 임직원 (회의실 예약자) — 별도 부서 구분 없이 누구나 예약 가능
- 관리자(총무팀 등) — 회의실 정보 관리, 전체 예약 현황 관리/취소

## 5. 핵심 기능 (MVP 범위)

### F1. 예약 현황 대시보드 (메인 화면)
- 회의실 1~6번을 컬럼(또는 행)으로, 시간대(예: 09:00~18:00, 30분/1시간 단위)를 축으로 하는 **그리드 타임테이블**
- 날짜 선택(기본값: 오늘) → 해당 일자의 회의실별 예약 현황 표시
- 예약된 칸: 회의제목 / 예약자 / 부서 노출, 색상으로 "예약됨" 표시
- 빈 칸: "예약 가능" 표시, 클릭 가능
- 실시간 반영: 다른 사용자가 예약/취소 시 화면 자동 갱신 (Supabase Realtime)

### F2. 빠른 예약 (원클릭 예약)
- 그리드에서 빈 시간 칸 클릭 → 예약 모달 오픈
  - 사전 채움: 회의실, 날짜, 시작시각(클릭한 칸 기준)
  - 입력 항목: 종료시각, 예약자명, 부서, 회의제목
- 저장 시 서버(DB) 단에서 동일 회의실·시간대 중복 여부 검증 후 확정
- 성공 시 그리드에 즉시 반영, 실패(중복) 시 사유 안내 후 재입력 유도

### F3. 예약 목록/검색
- 날짜/회의실/부서/예약자 기준 필터링이 가능한 리스트 뷰
- 예약 상세 확인, 취소 처리 (상태를 "취소"로 변경, 데이터는 삭제하지 않고 이력 보존)

### F4. 회의실 정보 보기
- 회의실별 수용인원, 층, 보유장비, 비고 확인 (팝오버 또는 별도 페이지)
- 그리드에서 회의실명 클릭 시 상세 정보 표시

### F5. 관리자 기능
- 회의실 마스터 데이터(이름/수용인원/장비/비고) 등록·수정
- 전체 예약 강제 취소/수정 권한

## 6. 화면 정의 (초안)

| 화면 | 설명 |
|---|---|
| 대시보드 (`/`) | 날짜 선택 + 회의실×시간 그리드, 클릭 예약 진입점 |
| 예약 모달 | 그리드에서 빈 칸 클릭 시 오버레이로 표시되는 예약 입력 폼 |
| 내 예약/전체 예약 목록 (`/reservations`) | 필터 가능한 테이블, 취소 액션 |
| 회의실 관리 (`/admin/rooms`) | 관리자 전용, 회의실 CRUD |

## 7. 데이터 모델 (Supabase / PostgreSQL 초안)

### `rooms`
| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | smallint (PK) | 1~6 |
| name | text | 예: "1번 회의실 (소회의실 A)" |
| capacity | int | 수용인원 |
| floor | text | 층 |
| equipment | text[] | 보유장비 (배열로 정규화) |
| note | text (nullable) | 비고 |

### `reservations`
| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | uuid (PK, default gen_random_uuid()) | 내부 식별자 |
| reservation_code | text (unique) | 표시용 코드 (예: B2026071), 자동 채번 |
| room_id | smallint (FK → rooms.id) | |
| booker_name | text | 예약자명 |
| department | text | 부서 |
| title | text | 회의 제목 |
| reservation_date | date | 예약일자 |
| start_time | time | 시작시각 |
| end_time | time | 종료시각 |
| status | text (`confirmed` \| `cancelled`) | 기본값 `confirmed` |
| created_at | timestamptz (default now()) | |

**중복 예약 방지**: `status = confirmed`인 건에 한해 동일 `room_id` + `reservation_date`에서 `[start_time, end_time)` 구간이 겹치는 예약을 막는 제약 필요.
- PostgreSQL `EXCLUDE USING gist` (btree_gist 확장 + tsrange/timerange) 사용을 권장
- 또는 애플리케이션 레벨 검증 + DB 트랜잭션으로 겹침 재확인 (경합 상황 대비 이중 방어 권장)

## 8. 비즈니스 규칙

- 예약 가능 시간: 평일 09:00~18:00 (정책 확정 필요, 8-1 이슈 참고)
- 최소 예약 단위: 30분 또는 1시간 (정책 확정 필요)
- 과거 시간대는 예약 불가
- 취소된 예약은 삭제하지 않고 `status = cancelled`로 이력 보존
- 동일 회의실·시간대 중복 예약 불가 (F2, 7장 참고)

## 9. 정책 확정 필요 이슈 (Open Questions)

1. **로그인 여부**: 이름/부서를 매번 텍스트로 입력할지, 사내 계정으로 로그인 후 자동 채움할지
2. **5번(대회의실) 우선 배정, 6번(스튜디오) 전용 목적**: 시스템적으로 예약 제한을 걸지, 단순 안내 문구만 노출할지
3. **예약 가능 시간대/최소 단위**: 09~18시 고정 여부, 30분/1시간 단위 여부
4. **예약 취소 권한**: 본인만 취소 가능한지, 관리자도 임의 취소 가능한지
5. **동시성 처리**: 두 사용자가 같은 빈 칸을 동시에 클릭했을 때 UX (선점 실패 안내 방식)

## 10. 비기능 요구사항

- **실시간성**: 예약/취소 발생 시 다른 사용자 화면에 수 초 이내 반영 (Supabase Realtime 구독)
- **반응형**: PC 기준 우선 설계, 모바일에서도 그리드 조회 가능하도록 반응형 대응
- **성능**: 회의실 6개 × 하루 시간대 조회는 소규모 트래픽이므로 별도 캐싱 없이 실시간 쿼리로 충분

## 11. 기술 스택 (제안)

- **DB/백엔드**: Supabase (PostgreSQL, Realtime, Row Level Security, Auth 선택적 도입)
- **프론트엔드**: 정책 결정 후 확정 (예: 정적 페이지 + Supabase JS Client, 또는 React 기반 SPA)

## 12. 향후 로드맵 (MVP 이후)

- 사내 계정 로그인 연동 (`auth.users` ↔ `reservations.booker_name` 매핑)
- 반복 예약(주간 정기 회의) 기능
- 예약 알림 (이메일/슬랙 연동)
- 회의실 이용률 통계 대시보드
- 부서별/개인별 예약 한도 설정
